## 一、pb_sync_manager.py — 双端口 SSH 隧道

### 背景

PocketBase 跑在服务器 `127.0.0.1:8090`，通过 SSH 隧道映射到本地 `18090`。
任务 HTTP 服务 `focusflow_tasks_server.py` 跑在服务器 `127.0.0.1:18091`，原先本地无对应映射。

### 解决

SSH 隧道增加第二个 `-L`：

```python
TASKS_REMOTE_ADDR = "127.0.0.1:18091"
TASKS_LOCAL_PORT  = 18091
TASKS_URL = f"http://127.0.0.1:{TASKS_LOCAL_PORT}"

# tunnel flags (excerpt)
"-L", f"18090:127.0.0.1:8090",
"-L", f"18091:127.0.0.1:18091",
```

新增方法：

```python
def get_today_tasks(self) -> list[dict]:
    """Fetch today's actionable 🍅 tasks from focusflow_tasks_server."""
    try:
        resp = requests.get(f"{TASKS_URL}/focusflow/today-tasks", timeout=5)
        return resp.json().get("tasks", [])
    except Exception:
        return []
```

## 二、main.py — 云端任务下拉菜单接入

### 问题（原有两个致命断层）

1. UI 下拉菜单仍读取本地 SQLite，不感知云端任务
2. `kr_ref` 没有传入 `record_session()`，PocketBase 同步记录缺失 KR 关联

### 解决

新增状态字段：

```python
self.current_kr_ref: str = ""
self._cloud_task_map: dict = {}   # display_str -> task dict
self._cloud_projects: dict = {}   # project_name -> [display_str list]
```

`_refresh_cloud_tasks()` 后台线程拉取，按 `project` 字段分组填充两个下拉框：

```python
def _refresh_cloud_tasks(self) -> None:
    def _fetch():
        tasks = self.pb_sync.get_today_tasks()
        if not tasks:
            return
        by_project: dict = {}
        display_map: dict = {}
        for t in tasks:
            proj = t.get("project", "FocusFlow")
            est  = t.get("pomodoros_est", 0)
            display = f"{t['task']} ({est}🍅)" if est else t['task']
            by_project.setdefault(proj, []).append(display)
            display_map[display] = t
        def _update():
            self._cloud_task_map = display_map
            self._cloud_projects = by_project
            cloud_proj_list  = list(by_project.keys())
            local_projects   = self.data_manager.get_all_projects()
            merged = cloud_proj_list + [p for p in local_projects if p not in cloud_proj_list]
            self.ui.project_combobox.configure(values=merged)
            if cloud_proj_list:
                self.ui.project_var.set(cloud_proj_list[0])
                self.ui.set_task_list(by_project[cloud_proj_list[0]])
                self.ui.task_var.set('')
        self.root.after(0, _update)
    threading.Thread(target=_fetch, daemon=True).start()
```

`on_project_selected()` 优先查 `_cloud_projects`；
`on_task_selected()` 查 `_cloud_task_map` 提取 `kr_ref` 并写入 `self.current_kr_ref`。

两处 `record_session()` 调用均补上 `kr_ref=self.current_kr_ref`。

### 坑 1：`_get_clean_task_name()` 未处理 `(N🍅)` 后缀

display 字符串格式为 `"任务名 (2🍅)"`，原来的 clean 函数不识别，导致写入 DB 时带着括号。

**修复：**

```python
m = re.match(r"^(.*) \(\d+🍅\)$", task_name)
if m:
    return m.group(1)
```

### 坑 2："☁️ Cloud" 标签不必要

原本用 `"☁️ Cloud"` 作为分组标签区分云端任务，但所有任务都是云端同步的，标签多余。

**修复：** 直接使用服务端返回的 `project` 字段（`Q1·职业事业`、`Q2·财富投资`、`Q4·智力技术`）作为分组 key，不再添加任何额外前缀。

## 四、Project 名称 Emoji 不一致修复

### 问题

迁移后 DB 里 project 名称带尾部 emoji（`Q1·职业事业 ⚔️`），
而服务端 `focusflow_tasks_server.py` 返回的 `project` 字段是干净名称（`Q1·职业事业`）。
两者 key 不匹配 → 下拉框出现重复分组。

### 解决

直接对 DB 执行 SQL，去掉尾部 emoji：

```sql
UPDATE projects SET project_name='Q1·职业事业' WHERE project_name='Q1·职业事业 ⚔️';
UPDATE projects SET project_name='Q2·财富投资' WHERE project_name='Q2·财富投资 💰';
UPDATE projects SET project_name='Q4·智力技术' WHERE project_name='Q4·智力技术 🧠';
-- 同步更新 tasks / sessions 表
```

## 五、Est vs Done 番茄数审计

### 结论

- 本地 DB `estimated_pomodoros` 是 **项目级别** 的累计估算（Q1 = 12、Q2 = 8、Q4 = 20），非任务级别
- WEEKLY_BRIDGE.md `note: 🍅 N` 是 **当日任务级别** 的估算（≤3 per task，由 `upsert_task.py` 工具层强制）
- `done_pomodoros = 0` 是因为 session 记录挂在 WEEKLY_BRIDGE 任务名下，与 DB 父任务名不对应 — 属于两套独立系统，**无需同步**

两套系统设计定位不同：
- 本地 DB sessions → 历史统计 / Dashboard
- WEEKLY_BRIDGE → 当日计划 / GoalSifter Bot 驱动

## 六、PocketBase Session 同步 + 飞书奖励机制调试

### 背景

番茄完成后期望触发：
1. 本地 session → PB `ff_sessions` → `reward.pb.js` hook → 飞书消息
2. `ff_daily_stats` 当日计数 +1

测试后两者均未触发。

### 坑 1：`_connected` 初始化失败后永不重试

`PBSyncManager` 启动时若 SSH 隧道认证失败，`_connected = False` 被写死，整个运行期间不再重试。番茄完成时走 offline queue，queue 只在下次成功启动时 flush。

**修复：** `_connect_bg()` 改为 `_reconnect_loop()`，每 60s 重试直到连上：

```python
def _reconnect_loop(self):
    while not self._connected:
        self._try_connect()
        if not self._connected:
            time.sleep(60)
```

### 坑 2：`local_id` 为空被 PB schema 拒绝

`record_session()` 中 `local_id = session_data.get("session_id", "")` 为空字符串时，PB 返回 `validation_required: Cannot be blank`。

**修复：** fallback 生成 UUID：

```python
"local_id": session_data.get("session_id", "") or str(uuid.uuid4()),
```

### 坑 3：`$os.exec` 在 PocketBase v0.28 JSVM 里不执行命令

`reward.pb.js` 用 `$os.exec('/root/.openclaw/send_reward.sh', [...])` — hook 日志显示 `exec returned` 但脚本根本没有运行（用写文件验证）。返回值是命令参数的字符串拼接，不是执行结果。**`$os.exec` 在此版本 Goja 运行时里是 no-op 或行为未定义。**

尝试 `nohup ... &` 后台方案：脚本本身可运行，但 PB cgroup 内的 nohup 子进程在 shell 退出后被 kill（`KillMode=control-group`，实测 reward.log 无新增）。

### 解决：tasks server 新增 `/focusflow/reward` 端点 + `$http.send()`

`$http.send()` 是 PocketBase JSVM 官方支持的 HTTP 客户端，100% 可用。

**focusflow_tasks_server.py** 新增 `do_POST` + `_send_reward()`：

```python
def _send_reward(data):
    # 组装消息，subprocess.run 调用 send_reward.sh
    ...

def do_POST(self):
    if self.path == '/focusflow/reward':
        body = json.loads(self.rfile.read(...))
        threading.Thread(target=_send_reward, args=(body,), daemon=True).start()
        # 立即返回 200，subprocess 在后台跑
```

**reward.pb.js** 改用 `$http.send()`：

```js

    url:    'http://127.0.0.1:18091/focusflow/reward',
    method: 'POST',
    headers: {'content-type': 'application/json'},
    body:   JSON.stringify({task_name, kr_ref, count, is_breakthrough})
});
```

验证：PB log 出现 `[reward] HTTP sent`，飞书收到奖励消息 ✅

### 额外坑：多次 shell heredoc 写入 Python 源文件导致字面换行嵌入 f-string

`$PYEOF` 等 heredoc 通过 SSH 写入 Python 文件时，多层 shell 转义使 `\n` 变成字面换行，Python 报 `SyntaxError: unterminated string literal`。**解决方案：本地编辑文件，SCP 推送，绝对不要通过 SSH heredoc 写含 Unicode 字符串的 Python 文件。**
