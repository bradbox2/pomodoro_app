# 2026-07-13 GoalSifter 桌面契约实机联通 + 应用内连接设置

本轮把 FocusFlow 真正接上了已部署的 GoalSifter FocusFlow API,并补齐了应用内配置入口。

---

## 1. SSH 只读核实:服务端其实已部署

原以为服务端契约尚未上线(本地代码库检索不到匹配 `/api/v1/focusflow/` + Bearer + `:8000` 的服务)。SSH 到 VPS(`106.53.153.227`)只读核实后发现服务端**已在运行**:

- `127.0.0.1:8000` 由 `uvicorn skills.goalsifter_core.api_server:app` 监听(workspace-v3)。
- `GET /api/v1/focusflow/tasks` 无 token 返回 **401**,带正确 Bearer 返回 **200** —— 端点存在且受 `FOCUSFLOW_WEBHOOK_TOKEN` 保护。
- token 存于 VPS `/root/.openclaw/secrets/focusflow.env`。

本地代码不含该服务,是因为云端部署版本比本地新(openclaw"云端为准")。全程只读,未改动任何云端文件。

## 2. 契约差异与客户端适配

服务端 `GET /tasks` 返回 **`{"tasks":[...]}` 信封**,而客户端 `get_active_dw_tasks()` 原本只接受**裸列表**,会误报 `GoalSifterRemoteError(500)`。

按项目边界(GoalSifter 服务端是独立仓库、不在本项目改动范围),改客户端去适配:

- `goalsifter_client.py`:`get_active_dw_tasks()` 现在解包 `{"tasks":[...]}`,同时兼容裸列表(向后兼容测试夹具)。
- 另两个端点(`POST /pomo-events`、`POST /tasks` 创建)的响应结构与客户端**已一致**,无需改动。
- `tests/test_goalsifter_client.py`:新增服务端信封解包、空信封两个用例。

## 3. 本机配置写入 + 端到端验证

- SSH 从 VPS 读取 token,写入本机 `%LOCALAPPDATA%\FocusFlow\goalsifter_settings.json`:`ssh_host_alias = "openclaw"`、`bearer_token = <token>`、`local_port = 18000`。token 只落本机,符合设计红线(SSH 只建隧道,不由客户端读取 VPS 密钥文件)。
- 端到端验证:用真实 `GoalSifterClient` 起隧道 `ssh -N -L 18000:127.0.0.1:8000 openclaw`,认证拉取成功返回 1 个活跃 DW。应用内"刷新 GoalSifter 任务"即可见。

## 4. 应用内 GoalSifter 连接设置对话框

免去手改 JSON,并支持启动自动连接。

- `goalsifter_settings.py`:新增 `auto_connect: bool = False` 字段(纳入 load/save),以及可单测的 `update_connection()`(裁剪空白、端口转 int、持久化)。
- `ui_manager.py`:GoalSifter 标签新增"⚙ 连接设置"按钮(初始构建 + 刷新重建两处)。
- `main.py`:`open_goalsifter_settings()` 弹出对话框(SSH 别名 / Bearer token 掩码可切换 / 本地端口 / "启动时自动连接"复选框;"保存"与"保存并测试连接");保存后失效旧隧道以便按新设置重连;`__init__` 中 `root.after(900, self._maybe_auto_connect_goalsifter)` 在勾选且已配置时启动即刷新。
- `tests/test_goalsifter_settings.py`:新增 `auto_connect` 往返、端口存为 int 两个用例。

## 5. 设计确认:KR 未被映射为本地 Project

针对"把服务器端 OKR 的 KR 映射为 Project 是否受本次改动影响"的疑问,核实数据层后确认**不受影响**,且现行模型本就不把 KR 当 Project:

- `pomodoro_data_manager.py:upsert_goalsifter_focus_item` 把远端 DW 存入**隐藏合成项目** `__goalsifter__:<task_id>`(按 task_id 键),`kr_ref` 仅存为只读 `context_label`。
- `get_local_project_summaries` 以 `NOT GLOB '__goalsifter__:*'` 把合成项目排除出用户可见 Project 列表;`merge_local_projects` 拒绝对其操作。
- 跨系统关联只用 `goalsifter_task_id`,不用 KR / 任务名。
- 旧的 KR→Project 映射(废弃的 `cloud-only/focusflow_tasks_server.py` 的 `kr_to_project()`、`scripts/migrate_projects.py` 的 OKR 改名)已于 2026-07-12 重构中被此解耦模型取代;遗留的 OKR 风格项目名现在只是静态个人标签。

本轮改动(信封解包 + 设置对话框)均未触碰这套逻辑。
