## 8. 修复:刷新 GoalSifter 报 WinError 10061(隧道就绪竞态)

**现象:** GoalSifter 标签点「刷新 GoalSifter 任务」报 `[WinError 10061] No connection could be made`。

**根因:** 隧道启动竞态。`main.py` 三处用隧道的流程(refresh / outbox sync / create-and-bind)都是先 `start_tunnel()` 拉起 `ssh -N -L 18000:127.0.0.1:8000 openclaw`,再**盲等固定 350ms** 就发 HTTP。SSH 建链(到 VPS 的 TCP + 认证 + 通道)常 > 350ms,请求发出时本地 `:18000` 还没 listen → 连接被拒 → 客户端 `URLError` 包成 `GoalSifterRemoteError(0, "…tunnel unavailable: [WinError 10061]…")`,UI 直接显示裸错误。无就绪检测、无重试;典型现象是第二次点击才成功。

**修复:** 用**非阻塞就绪轮询**取代盲等。

- `goalsifter_client.py`:新增 `is_tunnel_ready(timeout=0.5)`——对 `127.0.0.1:local_port` 做轻量 TCP 探测(不发 HTTP)。
- `main.py`:新增 `_await_goalsifter_ready(on_ready, on_timeout, attempts=20)`——在 Tk 事件循环上每 250ms 探测一次(最多 ~5s),就绪即 `on_ready()`;若 ssh 进程已退出或超时,`on_timeout(msg)` 给出可操作提示(而非裸 10061)。三处统一改走此轮询器;隧道判活分支也路由过来(就绪即刻返回)。create-and-bind 把实际创建逻辑抽成 `do_create` 闭包作为 `on_ready`,避免自重入死循环。
- `tests/test_goalsifter_client.py`:新增 `is_tunnel_ready` 的 True(临时 listen 端口)/ False(无监听端口)两个用例。

**验证:** `pytest` → **51 passed**(49 + 2);两个改动文件 `py_compile` 通过。手动预期:冷启动首点即显示「正在连接…」→ 数百毫秒后加载活跃 DW,不再报 10061;别名故意配错则 ~5s 后给出「隧道连接超时/进程已退出」提示。
